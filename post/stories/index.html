User Stories As Tools For Thought

(But not for planning)

December 20, 2024

Last week I wrote about novelty in software engineering, and its relation to complexity. Now most of us are familiar with thinking tools for managing complexity such as documenting requirements, diagramming, statistics and so on. But novelty requires us to be creative, to invent new solutions.

So this week I want to share one of the most helpful tools I’ve found for creative thought in software: user stories. And I don’t mean “tickets in Jira” - in fact I think user stories make for a poor planning tool, which I’ll explain later.

What are user stories?

What I mean by user stories is this: a text document which describes the different types of software users, and lists their needs as user stories.

Let’s imagine we’re creating an app that lets customers trade stocks, like eTrade. First let’s try to enumerate the different types of users. There are many ways to do this. We could contrast internal users (reps, admins, developers) and external users (customers, suppliers, service providers). We could segment customers by size, such as SMB and Enterprise users.

User identity is not enough however. In the user story document, the key criterion for different user types is that they have different needs, which are expressed as stories. So “day trader” and “investor” are both external customers, but they have quite different needs:

Day Trader

  • I want to trade high volume stocks with sophisticated trading strategies to make short term profits.
  • I need to see the order book in order to size my trades.
  • I need low latency trade execution to minimize drift between my requests and price movement. …

Investor

  • I want to buy stocks at the best available price, for the lowest fees.
  • I want the software to auto-balance my portfolio amongst different asset classes to mitigate risk.
  • I must be notified of relevent investor relation events and my voting options. …

Usually the product manager is the best person to draft the user story document, but tech and other stakeholders should participate. The user stories should be kept up to date to match the team’s understanding of the software as it is developed. It can also be used as a simple planning control to prioritize stories.

Why user stories help with novelty

The process of identifying relevant user types and describing their needs improves the team’s understanding of the problem the software is meant to address. With novel domains in particular, there is less prior art to copy, and the team is likely different understanindg/opinions ….

Discussing the overlapping and conflicting needs of users helps the team prior

Stories

  • Natural language, short
  • Different user perspectives more comprehensive descriptions
  • Encourage participation
  • Good for empathy, the why
  • Focus on value not details
  • Comprehensible by everyone

Why user stories are a poor planning tool

If user stories succinctly articulate users' needs, what are the chances that they are also the right unit of work to be used for coordinating a software project? That would be an amazing coincidence.

And yet that is the common claim.¹ And when you wed user stories to planning and there is a conflict, what do you do?

If a story won't fit in an iteration, you can split the story into two or more smaller stories

— User Stories Applied, Mike Cohn,

In other words, contort user stories to fit into sprints.

User stories are great, but the users' perspective isn’t the whole story. What about project tasks addressing things hidden from users, or only tangentially related to their needs? So called “story-less” tasks like optimizations, fixing bugs and upgrading software versions. These don’t belong to stories, yet they must be done.

And finally, what happens when you need to communicate the team’s understanding of users to project stakeholders, or onboard a new team member? A bucket of Jira tickets isn’t a good communication medium for that.²

The answer is not to sacrifice user stories for planning' sake. And that doesn’t mean not “doing” Agile. In fact neither the Agile Manifesto nor Scrum Guide mention user stories at all.³

Use stories to develop and communicate a shared understanding of the software’s users. Use tickets of work tasks to coordinate the implementation. They shouldn’t be the same thing.
